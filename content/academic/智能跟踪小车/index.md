这门课程设计的命题很直接：做一个能"认人、跟人、听懂人说话"的机器人。但一旦开始动手，三个矛盾马上浮出水面——嵌入式芯片算力不够跑大模型；纯云端方案网络一断就失能；视觉和语音两条链路耦合在一起互相抢资源。

解决思路是把这三个问题转化成一个架构问题：**不同任务按实时性和计算量分配给不同硬件**，让每个设备只做它最擅长的事。

## 总体架构

系统由四层组成：

```
云端大模型
  └── 复杂语义理解、图文联合推理、高层决策
树莓派（语音边缘端）
  └── 音频采集/降噪、ASR 上传、TTS 播放、MQTT broker
K230（视觉边缘端）
  └── 人脸检测、目标追踪、云台控制
ESP32-S3（底层执行端）
  └── 舵机/电机控制、传感器采集、屏幕显示、安全保护
```

各节点通过局域网 MQTT 通信，云端输出统一的 JSON 指令（包含 `emotion`、`action`、`voice` 三个字段），分别驱动两个终端完成屏幕表情切换、底盘动作执行和语音播报。

这种分工的好处是：ESP32-S3 不跑任何大模型，只做实时控制；K230 专注视觉计算；语音高采样率的音频处理全交给树莓派的 Linux 系统；云端只负责"思考"，不碰硬件。

## 人脸识别与目标追随

视觉链路由 K230 承担。K230 以摄像头采集画面，运行轻量化人脸检测算法，输出目标中心偏移量（pan/tilt）和边界框信息，通过 MQTT 发布到 `qrs/robot01/track` topic。

ESP32-S3 订阅该 topic 后，把画面偏差转换为航向误差，通过 **差速 PID 控制**调整左右轮速度：

- 目标偏左 → 右轮加速，左转修正
- 目标偏右 → 左轮加速，右转修正
- 目标居中后逐步降低控制量，避免超调抖动

为了让追随过程更平稳，追随逻辑做成了状态机：

| 状态 | 触发条件 | 行为 |
|------|---------|------|
| TRACK_LOCK | 目标稳定出现 | 闭环跟随 |
| Short-term | 目标短暂丢失（< 800 ms） | 低速搜索 |
| Long-term | 目标长时间丢失 | 停止跟随，重新搜索 |

PID 参数支持通过 MQTT 在线更新并持久化到 NVS，调试时无需反复重烧固件。

## 语音智能交互

语音链路完全由树莓派独立承担，与 ESP32-S3 零耦合。

麦克风（INMP441）采集音频后，经树莓派降噪预处理，通过讯飞 ASR 转为文本，再按三级策略分流：

1. **急停类**（"停止""危险""急停"）：本地规则直接生成停车动作，不走云端，延迟最低
2. **常用控制类**（"前进""左转""开始跟随"等）：轻量解析后转换为标准控制指令
3. **开放问答类**（"你看到了什么""今天天气怎么样"）：调用云端大模型，返回结构化 JSON

云端返回结果后，树莓派解析 `voice` 字段，通过 MAX98357A 功放 + 腔体喇叭播报语音回复；ESP32-S3 同时解析 `action` 和 `emotion` 字段，执行对应底盘动作和屏幕表情切换。

底盘动作白名单（`forward`、`backward`、`turn_left`、`turn_right`、`stop` 等）限制了大模型输出范围，防止模型生成不可控指令。

## 自动避障与本地安全保护

安全保护逻辑运行在 ESP32-S3 本地，不依赖网络：

- 超声波测距 < 15 cm：进入 `obstacle_hold`，停止前进
- 距离恢复到 > 22 cm：解除限制
- 目标丢失 > 800 ms：停止追随，不再按旧目标继续运动
- 人脸框过大（目标过近）或云台俯仰角超限：停止底盘追随动作
- MPU6050 检测到姿态异常：触发紧急限制

这层本地保护在云端断线或网络波动时依然有效，是系统安全的最后一道防线。

## 屏幕表情与拟人化反馈

ST7735 128×160 液晶屏基于 LVGL Canvas 自绘渲染，采用赛博朋克暗色调风格，根据机器人实时状态动态切换：

- **FOLLOWING**：星眼微笑
- **SEARCHING**：螺旋眼 + 左右扫描线动画
- **BLOCKED / TOO CLOSE**：菱形眼危险警示
- **HOLD**：待机状态

每种状态配合底部文字标注，与语音播报、底盘动作形成多模态一致的交互反馈。

## 通信架构

所有节点通过树莓派上的 Mosquitto MQTT broker 交换数据，核心 topic 如下：

| Topic | 发布方 | 内容 |
|-------|--------|------|
| `qrs/robot01/track` | K230 | 视觉追踪结果（偏移量、边界框） |
| `qrs/robot01/cmd/request` | 树莓派 | 动作请求（含 request_id） |
| `qrs/robot01/cmd/ack` | ESP32-S3 | 执行回执 |
| `qrs/robot01/gimbal/request` | ESP32-S3 | 云台动作请求 |

`request_id` 字段用于把语音端发出的请求和 ESP32 的执行回执对应起来，便于日志追踪和故障定位。

此外还有 Web 仪表盘和 Android App 接入同一 MQTT 系统，用于 PID 参数在线调整、状态可视化和手动控制。

## 遇到的主要问题

**追随抖动**：初版直接用画面偏移量控制底盘，视觉检测结果本身有抖动，加上电机惯性，动作幅度很大。改用状态机 + 人脸尺寸滤波 + 转向优先策略（偏差大时先降前进速度完成转向）后明显改善。

**多任务阻塞**：ESP32-S3 上图像采集、舵机控制、屏幕刷新、MQTT 通信并发运行，初期容易相互阻塞。拆分成独立 FreeRTOS 任务、用队列和信号量传递数据之后稳定下来。

**舵机电源波动**：舵机启动瞬间电流大，会导致主控复位。在舵机供电端并联 1000 μF 大电容滤波后解决。

**语音噪声**：室内环境噪声明显影响识别率，在树莓派端增加音频降噪预处理后准确率提升。
