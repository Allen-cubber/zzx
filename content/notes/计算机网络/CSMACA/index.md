在 IEEE 802.11 无线局域网标准中，CSMA/CA 是分布式协调功能（DCF）的基础。以下是该协议核心机制的严谨推演。

### 一、 为什么必须是 CA（碰撞避免）？—— 物理层与拓扑限制

在无线信道中，直接套用有线以太网的 CSMA/CD（碰撞检测）在工程实现上是不可行的，主要受制于两个物理原因：

1. **收发硬件的动态范围限制**：无线网卡在全功率发送射频信号时，其发射功率远大于到达接收天线的其他微弱信号（相差可能达百万倍）。这就导致接收器在发送期间会因自身信号过载而“致盲”，**无法做到全双工的“边发边听”**，自然无法即时检测碰撞。
    
2. **隐蔽站问题（Hidden Terminal Problem）**：由于无线电波的衰减和障碍物遮挡，节点 A 和节点 C 都能与节点 B 通信，但 A 和 C 互相处于对方的无线电覆盖范围之外。A 发送时 C 听不到载波，C 误认为信道空闲也发起发送，导致在 B 处发生不可见碰撞。
    

鉴于无法检测碰撞，802.11 MAC 层采用了**“尽量减少碰撞概率（CA）+ 肯定确认（ACK）”**的机制。

### 二、 CSMA/CA 的三大核心机制

为了实现“避免”，协议设计了三个精巧的时间与逻辑控制维度：

#### 1. 帧间间隔（IFS, InterFrame Space）体系

802.11 规定，所有的节点在发送数据前，必须先监听信道空闲一段指定的时间。为了区分不同帧的优先级，协议定义了多种长度的 IFS：

- **SIFS（短帧间间隔）**：最短的等待时间（例如 28 $\mu s$）。用于最高优先级的控制帧，例如 **ACK 帧**、CTS 帧等。这保证了接收方在回复 ACK 时，不会被其他试图发送普通数据的节点抢占信道。
    
- **PIFS（点协调功能帧间间隔）**：中等长度，主要用于 PCF（点协调功能）模式下的无竞争接入，在常规 DCF 中较少使用。
    
- **DIFS（分布式协调功能帧间间隔）**：最长的等待时间（例如 128 $\mu s$）。用于普通的异步数据帧和管理帧发送。任何节点想发送普通数据，**必须至少监听到信道连续空闲 DIFS 时间**。
    

#### 2. 物理与虚拟双重载波监听

- **物理载波监听**：由物理层（PHY）负责，直接检测空间射频能量和特定的调制信号同步头，判断信道是否真正被占用。
    
- **虚拟载波监听与 NAV（网络分配向量）**：这是 MAC 层的逻辑控制。802.11 数据帧的 MAC 首部中含有一个 `Duration`（持续时间）字段，指明了本次传输（包括后续的 ACK）还需要占用信道多长时间。
    
    - 其他节点监听到这个帧后，即使该帧不是发给自己的，也会读取 `Duration` 的值，并将其更新到自己内部的一个倒计时器——**NAV（Network Allocation Vector）**中。
        
    - **只要 NAV > 0，节点就认为信道处于“逻辑忙”状态**，无论物理层是否检测到射频能量，都绝不发送数据。这极大降低了碰撞概率。
        

#### 3. 二进制指数退避算法（Binary Exponential Backoff）

当节点有数据要发送时，如果信道正忙，或者刚刚空闲了一个 DIFS，为了防止多个憋了很久的节点同时发送导致彻底崩溃，必须引入随机退避。

- 节点从**竞争窗口（Contention Window, CW）**中服从均匀分布随机选择一个整数 $r$，范围是 $[0, CW-1]$。
    
- 计算退避时间：$T_{backoff} = r \times \text{SlotTime}$（时隙长度，物理层定值）。
    
- 节点启动退避计时器：
    
    - 当信道**空闲**时，计时器递减。
        
    - 当信道**忙碌**时，计时器**冻结（挂起）**，保持当前值。
        
    - 当信道再次空闲经过 DIFS 后，计时器从挂起的值**恢复递减**。
        
- **当计时器递减到 0 时，节点立刻发送整个数据帧。**
    
- 如果发生碰撞（未收到 ACK），则 $CW$ 加倍（直至最大值 $CW_{max}$），重新选择更大的退避时间进行重传。
    

### 三、 完整传输状态机演示

假设节点 A 要向节点 B 发送数据帧，严格的时序流程如下：

1. **初始监听**：A 检测信道。如果信道一直空闲，且持续时间达到了 DIFS，A **直接发送**数据帧（不退避）。
    
2. **接收与确认**：B 完整接收数据帧，进行 CRC 校验。若无误，B 等待一个 **SIFS** 时间后，立刻向 A 发送 **ACK 帧**。
    
3. **退避介入（信道忙时）**：如果 A 初始监听时发现信道忙（或在其等待 DIFS 期间信道变忙），A 必须推迟发送。
    
4. **执行退避**：A 等待信道重新变为空闲，且保持空闲时长达到一个 DIFS 后，开始执行上述的**随机退避算法**。
    
5. **抢占失败的处理**：如果在 A 的倒数过程中，节点 C 的计时器先到 0 并开始发送，A 必须立刻**冻结**自己的倒计时，更新 NAV，等 C 发送完且信道再空闲 DIFS 后，继续从冻结的值往下倒数。
    

### 四、 针对隐蔽站的扩展：RTS/CTS 握手机制

为了解决第一点提到的隐蔽站问题，特别是当发送的数据帧极长（长帧碰撞代价极大）时，MAC 层引入了**可选的信道预约机制**。

1. A 在倒数到 0 后，不直接发长数据帧，而是先发一个极短的控制帧：**RTS（Request To Send，请求发送）**。RTS 帧内包含了后续通信所需的总时间。
    
2. B 收到 RTS 后，如果同意接收，等待 SIFS 后广播一个 **CTS（Clear To Send，允许发送）**。CTS 同样包含通信所需总时间。
    
3. **关键点**：所有能听到 A 发送 RTS 的节点（即使听不到 B），以及所有能听到 B 发送 CTS 的节点（隐蔽站，听不到 A），都会据此**更新自己的 NAV**，在通信期间保持沉默。
    
4. A 收到 CTS 后，等待 SIFS，开始发送真正的数据长帧。B 收到后，再等待 SIFS，回复 ACK。
    

通过 RTS/CTS，CSMA/CA 成功将大面积的不可见碰撞风险，缩小到了仅在发送极短的 RTS 帧时才可能发生的极小概率事件中，极大提升了底层信道的吞吐量。
